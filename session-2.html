Walkthrough

What Git is and how to set it up.

Everything Ryan and Maroun covered in Session 2, written out for reference. If you missed the class or want a refresher on the concepts, work through these sections at your own pace.

Concept What is Git (and why should you care)?

Git is a version control system — it tracks every change you make to your code. Think of it like a detailed undo history that never goes away.

GitHub is the cloud platform where your Git projects (called repositories or repos) are stored and shared. It's where collaboration happens.

Why this matters for you:

Collaboration

Git is how you share code with engineers and other builders. When your prototype is ready for production, Git is the handoff mechanism.

Safety net

Every change is saved. You can always go back to a previous version. No more "I accidentally deleted everything" panic.

Parallel universes

Branches let you (and AI agents) explore different directions without affecting the main codebase. Pick the one you like, discard the rest.

Concept Commits — save points for your code

A commit is a snapshot of your code at a specific point in time. Think of it like a save point in a video game — you can always go back to it.

Each commit has a message that describes what changed. Good commit messages help you (and others) understand the history of the project.

"Add yellow background to hero section" is a good commit message. "Changes" is not.

You don't need to manually create commits — Claude Code will do this for you. Just tell it to commit your changes and it'll write a sensible message.

Concept Branches — parallel copies of your code

A branch is a copy of the codebase that you can change without affecting the original.

The main branch is the central, "official" version of the code. When you want to work on something, you create a new branch off of main, make your changes there, and then bring those changes back when you're ready.

For prototyping

You can create a branch, experiment to your heart's content, and it only lives on your computer until you push it. Nobody else sees your work until you're ready.

For collaboration

If Jen and Lizzie are both working on the same project, they can each have their own branch. When Lizzie finishes, she pushes to GitHub. Jen pulls those changes down and keeps working.

To create a branch, just tell Claude Code: "Create a new branch called my-feature". It handles the Git command for you.

Concept Push & Pull — syncing local and remote

There are two copies of your code: what's on your computer (local) and what's on GitHub (remote). Push and pull keep them in sync.

Push

Upload your local changes to GitHub. After you've made commits, push them so others can see your work.

Pull

Download the latest changes from GitHub to your computer. Do this to get changes your collaborators have pushed.

⚠️ Key distinction: "Pull" (downloading changes) is different from "Pull Request" (a formal review process). They just share a name — don't let that trip you up.

You'll hear the words remote and origin — for now, they both mean GitHub. There's a world where they can be different, but we're not going there yet.

To check if you're up to date, just ask Claude Code: "Am I behind the remote branch?"

Concept Pull Requests — the handoff to engineering

A pull request (PR) is a formal request to merge your branch into the main codebase. But more importantly, it's a conversation.

When you create a PR, you're saying: "Hey engineering, here's what I've built. Let's talk about it."

Ownership handoff

When you create a PR, ownership of the code transfers to the engineer. They review it, contribute to it, discuss changes, and ultimately approve and merge it.

You don't merge

The engineer handles merging, QA, and the release pipeline. Especially in Corgi (our main codebase), there are CI/CD processes you don't need to worry about.

For prototypes: The PR process matters less. You're working in your own isolated environment and Claude Code handles branching for you. PRs become important when your work is heading to production.

Step 1 Install Git (Xcode command line tools)

Open Terminal and run:

xcode-select --install

This installs Apple's command line developer tools, which includes Git. If you've already done this, it'll tell you it's already installed — that's fine.

Verify it worked by running git --version. You should see a version number.

Step 2 Install GitHub CLI

The GitHub CLI (gh) lets you interact with GitHub from the terminal. Install it with Homebrew:

brew install gh

If you don't have Homebrew yet, Claude Code can install it for you — just ask.

⚠️ Watch out: If brew gives you a "command not found" error, you need to install Homebrew first. Ask Claude Code: "Install Homebrew for me."
Step 3 Authenticate with GitHub

Run this in your terminal:

gh auth login

You'll be asked a series of questions. Here's what to pick:

  • → Where do you use GitHub? → GitHub.com
  • → Preferred protocol? → HTTPS
  • → How to authenticate? → Login with a web browser

It will give you a one-time code and open a browser window. Enter the code, authorize the app, and you're connected.

⚠️ Watch out: Remember — mouse doesn't work in the terminal. Use arrow keys to navigate and Enter to select. If you're stuck, try the keyboard shortcut Option+Arrow to jump between words.
⚠️ Watch out: If you get "command not found" after installing gh, try closing and reopening your terminal window first.
Step 4 Verify your setup

Run this to confirm everything is working:

gh api user

This should show your GitHub username and profile info. If it does — you're all set.
